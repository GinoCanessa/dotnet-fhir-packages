# Incomplete release: 2026.722.1030

Release `2026.722.1030` is an incomplete historical release. The SDK package
was published, but the matching CLI package was not produced or published.
The version will not be backfilled; a later synchronized version must
supersede it.

## Release evidence

| Field | Value |
|-------|-------|
| Version | `2026.722.1030` |
| Tag | [`v2026.722.1030`](https://github.com/GinoCanessa/dotnet-fhir-packages/releases/tag/v2026.722.1030) |
| Source commit | [`b4b1a9c`](https://github.com/GinoCanessa/dotnet-fhir-packages/commit/b4b1a9c) |
| GitHub Actions run | [`29933893584`](https://github.com/GinoCanessa/dotnet-fhir-packages/actions/runs/29933893584) |
| Workflow result | Successful |
| `fhir-pkg-lib` | NuGet.org returned HTTP `200` for `2026.722.1030` |
| `fhir-pkg-cli` | NuGet.org returned HTTP `404` for `2026.722.1030` |

At investigation time, the latest SDK version was `2026.722.1030`, while the
latest CLI version remained `2026.622.1701`.

## Impact

SDK consumers could install the release, but `dotnet tool install` and
`dotnet tool update` could not obtain the matching CLI. CLI-specific fixes in
the release source therefore remained unavailable through NuGet.org, and the
repository's shared-version contract was violated even though the workflow
reported success.

## Cause

The hardened release workflow packed, qualified, published, and verified only
`fhir-pkg-lib`. It did not create an immutable `fhir-pkg-cli` candidate, so no
qualified CLI artifact exists for this version.

## Recovery decision

Do not publish `fhir-pkg-cli` `2026.722.1030` manually and do not rebuild the
old candidate. Publish a new version through the synchronized SDK/CLI workflow
after validation-only qualification succeeds. Once that replacement is
available:

1. Annotate the GitHub Release for `v2026.722.1030` as incomplete.
2. Deprecate `fhir-pkg-lib` `2026.722.1030` on NuGet.org in favor of the
   replacement synchronized version.
3. Keep this record as the immutable explanation for the version gap.
